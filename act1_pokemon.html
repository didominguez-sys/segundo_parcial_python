Actividad Opción 1: Crear una API de Colección de Pokémon

Esta actividad te permitirá simular una aplicación donde puedes "atrapar" y gestionar tus Pokémon favoritos utilizando FastAPI.

Objetivo: Desarrollar una API que permita añadir, listar y ver detalles de Pokémon.

Conceptos clave a aplicar:

  • Instalación y configuración básica1:
    • Abre tu terminal e instala FastAPI y Uvicorn. Recuerda que Uvicorn será el servidor para ejecutar tu aplicación.
    • Crea un nuevo directorio para tu proyecto y un archivo main.py.
    • Importa FastAPI y crea una nueva aplicación en main.py.
  • Ruta raíz (/):
    • Define una ruta de inicio para el método HTTP GET en tu aplicación1. Cuando alguien visite esta ruta, debería devolver un mensaje de bienvenida, por ejemplo, "¡Bienvenido a tu Pokedex personal con FastAPI!".
    • Recuerda ejecutar tu servidor usando Uvicorn con uvicorn main:app --reload en tu terminal para que los cambios se actualicen automáticamente.
  • Modelo de Datos con Pydantic (Pokemon):
    • Importa BaseModel de Pydantic.
    • Crea una clase Pokemon que extienda BaseModel. Esta clase representará la estructura de un Pokémon en tu colección. Debería tener al menos los siguientes atributos:
      • nombre (cadena, obligatorio).
      • tipo (cadena, por ejemplo, "Fuego", "Agua", "Planta").
      • nivel (entero, con un valor predeterminado como 1).
      • es_legendario (booleano, con un valor predeterminado como False).
    • La utilización de Pydantic hará que la validación de datos sea más sencilla y mejorará la documentación interactiva.
  • Lista de Pokémon (en memoria):
    • Crea una lista vacía global en tu main.py llamada pokedex para almacenar los Pokémon capturados.
  • Ruta para "Capturar" un Pokémon (POST):
    • Crea un nuevo punto final para añadir Pokémon (por ejemplo, /pokemon) que acepte solicitudes HTTP POST.
    • Este punto final deberá esperar un cuerpo de solicitud JSON que se ajuste a tu modelo Pokemon. Cuando recibas un Pokémon, añádelo a tu lista pokedex.
    • Puedes optar por devolver el Pokémon recién añadido o la lista completa de Pokémon después de la adición.
  • Ruta para "Listar" todos los Pokémon (GET):
    • Crea un punto final para listar todos los Pokémon (por ejemplo, /pokemon) que acepte solicitudes HTTP GET.
    • Este punto final simplemente devolverá la lista pokedex completa.
    • Aplica el response_model para asegurar que la respuesta se adhiera a un modelo Pydantic.
  • Ruta para "Ver" un Pokémon específico (GET con parámetro de ruta):
    • Implementa un punto final (por ejemplo, /pokemon/{pokemon_id}) donde pokemon_id sea un parámetro de ruta.
    • Manejo de Errores (HTTPException): Si el pokemon_id no existe, lanza una HTTPException con código 404.
    • Aplica también el response_model para asegurar que la respuesta del Pokémon individual se ajuste a tu modelo Pokemon.
  • Documentación Interactiva:
    • Visita /docs o /redoc en la URL de tu servidor FastAPI.
    • Explora y prueba cada uno de tus puntos finales directamente desde Swagger UI.

Analogía

Piensa en FastAPI como un chef muy eficiente y organizado para tu restaurante (tu aplicación web).

  • Cuando recibes un pedido (una solicitud HTTP a tu API), el chef (FastAPI) sabe exactamente qué plato preparar (qué función de ruta ejecutar)
  • Los Pydantic Models son como las recetas estandarizadas y validadas del chef. Aseguran que los ingredientes (los datos que envías o recibes) tengan la forma y calidad correctas, evitando desastres en la cocina (errores en tu aplicación).
  • La documentación interactiva (/docs) es como el menú detallado del restaurante, pero con la capacidad de probar cada plato antes de pedirlo.
